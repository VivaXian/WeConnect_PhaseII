# PhaseI_Reference.md

> 用途：给 VS Code / AI 设计模型理解 WeConnect Phase I 已有产品结构、页面心智、可复用交互模式。  
> 边界：这是参考文件，不是新需求文档。

---

## 1. Phase I 产品基线

Phase I 的核心价值是：**极简报修 + 报修进展查看 + 工单签字/评价**。

当前底部导航为：
1. 我的报修
2. 工单列表
3. 我的

Phase II 设计时应尽量复用 Phase I 已有用户心智和开发成果，不应重做一个全新产品。

---

## 2. Phase I 已有核心页面

### 2.1 登录 / 隐私确认
- 微信快捷登录
- 隐私政策确认
- 未注册手机号登录后自动创建飞利浦账号

### 2.2 扫码报修 / 设备识别
- 扫描设备二维码
- 自动带出设备信息
- 展示设备名称、医院、EQ、序列号
- 主要操作：极速报修
- 次要操作：咨询

### 2.3 报修表单
- 设备信息
- 是否造成伤害
- 是否停机
- 故障描述
- 附件上传：照片 / 视频 / 录音
- 报修联系人
- 提交

设计启示：Phase II 应保留低摩擦报修入口，不要在报修前增加不必要复杂度。

---

## 3. “我的报修” / 报修进展列表

“我的报修”不是简单历史记录，而是用户理解设备服务状态的核心入口。

已有内容包括：
- 扫码报修入口
- 搜索：设备名称 / 医院
- 报修记录列表
- 按月份分组
- 设备名称
- 医院 / 院区 / 科室 / location
- 当前服务状态
- 最新节点摘要
- 工程师信息
- 设备详情入口
- 报修详情入口

已有报修状态包括：
- 已报修
- 服务中
- 服务完成 / 已完成
- 已取消

设计启示：
- Phase II 不应弱化报修进展列表。
- 普通用户仍应主要通过这里查看自己的服务状态。
- Super User 可以扩展数据范围，但页面本质仍应是状态列表，不应变成重 dashboard。

---

## 4. 报修详情

已有报修详情包括：
- 报修编号
- 服务工程师
- 工单信息
- 维修进度时间线
- 关键节点：报修已收到、已分配工程师、备件已出库、服务完成等
- 设备历史报修

设计启示：
- Phase II 的 PM 结果、风险提醒、合同提醒也应解释“为什么提醒”和“与哪台设备有关”。
- 可延续时间线 / 状态节点的表达方式。

---

## 5. 设备详情

已有设备详情包括：
- 设备名称
- 医院
- 科室 / location
- EQ
- 序列号
- 修改备注
- 电话咨询
- 极速报修
- 设备历史报修

设计启示：
- 设备详情是 Phase II 扩展合同 / PM / 风险信息的重要承载页。
- 合同和 PM 信息应绑定在设备语境下展示。

---

## 6. 工单列表

工单列表不只是“待签字”，而是工单对象与收尾动作页。

已有工单类型包括：
- 维修
- 保养
- FCO
- 安装

已有工单状态 / 动作包括：
- 去签字
- 已签字
- 请求已失效
- 服务评价
- 下载 PDF
- 发送到邮箱

设计启示：
- Phase II 不应把工单 tab 命名为“待签工单”。
- “待签字”应该是状态 / 筛选项，而不是整个页面定义。
- Super User 可能看到较大范围工单，因此分类和筛选非常重要。

---

## 7. Phase I 筛选模式

### 7.1 报修列表筛选
- 搜索：设备名称 / 医院
- 报修状态：已报修、服务中、已完成、已取消
- 时间范围：近三个月、近六个月、近一年

### 7.2 工单列表筛选
- 搜索：设备名称 / 医院
- 工单状态：待签字、已签字、请求已失效
- 工单类型：维修、保养、安装、FCO
- 时间范围：近三个月、近六个月、近一年

设计启示：
- Phase II 应复用“搜索 + 状态 + 时间范围 + 类型”的筛选心智。
- Super User 可增加医院 / 院区、合同状态、PM 状态、风险状态等管理型筛选。

---

## 8. Phase I 复用原则

AI 设计时必须遵守：
1. 保留报修流程。
2. 保留报修进展列表的核心地位。
3. 保留工单列表作为工单对象 / 动作页。
4. 复用已有筛选模式。
5. 扩展设备详情，而不是创建脱离设备的合同中心。
6. 不要把 Phase II 设计成一个全新的复杂 dashboard。
7. 面向用户使用熟悉词：报修、设备、工单、保养、我的。
