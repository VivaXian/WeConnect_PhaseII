# IA_Proposal.md
# WeConnect Phase II｜信息架构提案

---

## 1. 当前 IA 评估

### 1.1 现状

当前代码实现的底部导航为：

```
工单 | 设备 | 消息 | 我的
```

普通用户的「工单」Tab 下存在二级 Tab：「工单 / 报修记录」。

### 1.2 问题清单

| # | 问题 | 影响 |
|---|------|------|
| 1 | 第一个 Tab 是「工单」而非「报修」 | 打断 Phase I 用户心智。Phase I 用户进入小程序首先找到的是报修进展，而不是工单签字。|
| 2 | 「消息」作为底部 Tab 独立存在 | 设计意图明确反对此做法。消息中心是弱提醒归档，不是核心任务入口，放底部导航会误导用户优先级认知。|
| 3 | 「报修记录」作为「工单」的二级 Tab | 逻辑颠倒。报修记录（修进展）是 Phase I 核心内容，应作为独立 Tab「报修」的主体，而非挂在「工单」下的子视图。|
| 4 | 「报修」入口消失 | 扫码报修和查看报修进展，是普通用户最高频的动作，不应被藏进二级 Tab。|
| 5 | PM schedule 无设备页强入口 | 月度 PM 计划目前只在消息中心可见，设计文件要求在「设备」Tab 有快速筛选入口。|
| 6 | Super User 的报修和工单页没有明确分离 | 目前 SuperUserServicePage 作为「工单」Tab 内容展示，实为报修记录，与工单签字动作混淆。|

### 1.3 评估结论

当前 IA **不合理**，需要重构 Tab 顺序和消息中心位置。这不是小调整，而是结构性修正。但现有页面组件（报修列表、工单列表、设备列表、消息中心、个人页）本身功能逻辑基本正确，只需重新编排到正确的 Tab 位置，不需要重写所有页面。

## 2. 推荐 IA

### 2.1 底部导航（两类角色相同结构）

```
设备 | 报修 | 工单 | 我的
```

#### Tab 设计原则
- **「设备」**：Phase II 核心对象，普通用户看自己的设备，Super User 看全院视图
- **「报修」**：Phase I 用户心智延续，扫码报修 + 报修进展
- **「工单」**：工单对象与收尾动作，不限于待签字
- **「我的」**：账户、权限、消息中心、设置——消息中心是内容，不是 Tab

### 2.2 各 Tab 职责

---
#### Tab 1：设备

**核心职责**：设备是 Phase II 复杂信息的核心容器，合同、PM、服务记录都绑定在设备下。

**普通用户**

```
设备页
├── 搜索：设备名称
├── 我的设备（无「最近查看」分类，统一展示相关设备）
└── 设备卡片
    ├── 设备名称（有自定义备注时以备注名为主，原名弱化）
    ├── 科室 / 所属医院
    ├── 异常状态 tag（只在异常时显示；正常设备不显示任何 tag）
    └── 不显示设备类型文字

设备详情（普通用户版）
├── 固定顶部区域（蓝色背景，可折叠）
│   ├── 设备名称（支持自定义备注，本地保存）
│   ├── 折叠区：EQ、序列号、科室/位置（用户可自填）
│   ├── 异常状态 chip（仅异常时显示）
│   └── 快捷按钮：电话咨询 / 极速报修
├── Tab 1：全部工单（维修/保养/安装/FCO 4类，按类型分组）
│   └── 权限提示：仅显示与本人相关的工单
├── Tab 2：报修记录（本人相关，倒序时间线）
│   └── 权限提示：仅显示与本人相关的报修
└── Tab 3：保养工单（时间线展示，含PM风险提示）
    └── 权限提示：仅显示与本人相关的保养工单

不显示：合同信息
```

**Super User**

```
设备页
├── 快速筛选（强入口）
│   ├── 全部设备
│   ├── 本月保养        ← PM schedule 强入口
│   ├── 即将到期
│   ├── 风险设备
│   └── 服务中
├── 搜索：设备名称 / 科室 / 院区
└── 设备卡片（展示全部状态标签）

设备详情（Super User 版）
├── 固定顶部区域（蓝色背景，可折叠）
│   ├── 设备名称（支持自定义备注）
│   ├── 折叠区：EQ、序列号、科室/位置（用户可自填）
│   ├── 异常状态 chip（已出保 / 即将出保 / 待验收 / 报修中 / PM高风险，仅异常时显示）
│   └── 快捷按钮：电话咨询 / 极速报修
├── Tab 1：全部工单（维修/保养/安装/FCO 4类）
├── Tab 2：报修记录（全渠道，倒序时间线）
├── Tab 3：保养工单（时间线展示，含PM风险提示；在保且有下次计划时不显示PM风险）
└── Tab 4：合同信息 [SU only]
    ├── 无「服务等级」概念
    └── 合同时间线倒序（Warranty → POS → CSA，最新在上）
```

#### Tab 2：报修

**核心职责**：延续 Phase I 核心价值，让用户快速报修并查看报修进展。

**普通用户**

```
报修页
├── 顶部：扫码报修 入口（固定）
├── 搜索：设备名称 / 医院
├── 状态筛选 chips：全部 / 服务中 / 待签字 / 已取消
├── 时间筛选：近三个月 / 近六个月 / 近一年
└── 报修记录列表（月份分组）
    └── 报修卡片
        ├── 设备名称 + 医院/科室
        ├── 当前状态标签
        ├── 最新进展节点摘要
        ├── 工程师信息（如已分配）
        └── 快速跳转：[设备详情] [报修详情]
```

**Super User**

```
报修页
├── 顶部：报修入口（可保留，但优先级低于普通用户）
├── 搜索：设备名称 / 医院 / 院区
├── 状态筛选：全部 / 服务中 / 已完成 / 已取消
├── 时间筛选：近三个月 / 近六个月 / 近一年
└── 报修记录列表（月份分组）
    └── 报修卡片（内容与普通用户相同，范围更广）

注意：Super User 报修页仍以列表为主，不做重 Dashboard。
```

#### Tab 3：工单

**核心职责**：工单对象与收尾动作页，不只是待签字。

**普通用户**

```
工单页
├── 搜索：设备名称 / 工单编号
├── 状态 tabs：全部 / 待签字 / 已完成 / 已失效
└── 工单卡片（月份分组）
    ├── 工单类型标签：维修 / 保养 / 安装 / FCO
    ├── 设备名称 + 医院
    ├── 工单编号 + 请求时间
    └── 动作区：
        ├── 待签字 → [去签字] 按钮
        ├── 已完成 → [评价服务] [下载 PDF] [发送邮箱]
        └── 已失效 → 状态展示，无动作
```

**Super User**

```
工单页
├── 搜索：设备名称 / 工单编号 / 医院
├── 状态筛选：全部 / 待签字 / 待处理 / 已完成 / 已失效
├── 类型筛选：全部类型 / 维修 / 保养 / 安装 / FCO
├── 时间范围：近三个月 / 近六个月 / 近一年
└── 工单卡片（同普通用户结构，范围更广）
```

#### Tab 4：我的

**核心职责**：账户与工具收口，消息中心在此作为入口，支持多设备聚合提醒。

```
我的页
├── Hero 区：头像 / 姓名 / 角色标签
├── 消息中心入口
│   ├── 行式入口：「消息中心」+ 未读数角标
│   └── 点击进入 P8 消息中心页
├── 账户信息
│   ├── 所属医院
│   ├── 科室
│   └── 手机号
├── 权限信息
│   ├── 当前角色（普通用户 / Super User）
│   ├── 权限范围说明
│   └── Super User：授权范围、有效期
├── （普通用户）申请升级为 Super User 入口
├── 常用设备管理
└── 设置
    ├── 消息通知设置
    ├── 帮助与反馈
    └── 退出登录

→ 消息中心页（P8，从「我的」进入）
    ├── 时间分组：今日 / 本周 / 更早
    ├── 消息类型分类：合同提醒 / 验收提醒 / PM计划 / PM风险 / 服务提醒
    ├── 已读 / 未读状态
    ├── 全部已读 + 批量删除功能
    ├── 多设备聚合消息：同条消息中列表展示多台设备
    └── 点击跳转：对应设备详情 / 工单详情
```

**消息推送策略**（重要）：
- **多设备合并原则**：将多台设备的相同类型提醒合并为一条消息，降低推送打扰频率
  - 合同到期提醒：同日期、同医院的多台设备 → 单条消息列表展示
  - PM 月度计划：该月所有 PM 计划 → 按医院/区域单条消息展示
  - PM 风险提醒：多台设备风险 → 按医院/区域单条消息展示
- **单次推送类型**（不合并）：工单待签字、权限升级通过等即时性强的提醒

## 4. 消息中心位置变更说明

### 核心原则
- **不放底部 Tab**：消息中心是弱提醒归档，放底部 Tab 误导用户优先级
- **不放顶部铃铛为主入口**：微信小程序顶部控件区空间有限，不可控
- **放在「我的」Tab 第一个区块**：位置显眼，带未读数角标，但不争抢底部导航位置
- **消息中心功能完整性不变**：分组、已读/未读、单条/批量删除、跳转功能保留

### 多设备聚合策略（新增）
推送消息时采用 **IB 级多设备聚合** 来减少推送频率：

| 消息类型 | 聚合规则 | 示例 |
|---------|---------|------|
| 合同风险 | 同日期、同医院的多台设备 → 一条消息 | 医院A有3台设备到期 → 单条消息列表展示 |
| PM月度计划 | 该月全部PM计划 → 按医院/区域单条消息 | 医院A该月有5台设备计划保养 → 单条消息含列表 |
| PM风险提醒 | 多台设备风险 → 按医院/区域单条消息 | 医院A有4台设备PM超期 → 单条消息含列表 |
| 验收提醒 | 同医院待验收设备 → 一条消息 | 医院A有2台设备待验收 → 单条消息列表 |
| 单次结果 | 不合并（即时推送） | 工单待签字、权限升级通过等 |

**用户感受**：
- 不推送"该设备保养已完成"的冗余通知
- 多设备提醒合并后，消息通知数量大幅降低
- 点击聚合消息中单台设备 → 跳转该设备详情，可查看完整信息

## 5. PM Schedule 强入口策略

PM schedule 不能只放在消息中心。推荐三个位置形成递进感知：

| 层级 | 位置 | 形式 |
|------|------|------|
| 发现 | 设备 Tab 快速筛选 chip | 「本月保养」chip，点击过滤本月有 PM 的设备 |
| 理解 | 设备详情 Tab 3 保养工单 | 时间线保养记录 + 上/下次保养日期 + 风险状态 |
| 提醒回看 | 消息中心 | PM 月计划提醒归档，点击跳转设备详情 |

## 6. 扫码入口策略

扫码报修和扫码绑定设备是全局功能，不局限于某一 Tab：

//- **全局扫码 FAB**：悬浮按钮，固定于底部导航上方，两类角色始终可见
//- 点击展开菜单：「扫码报修」「扫码绑定设备」
- 此设计替代将扫码入口仅放在报修 Tab 顶部的做法，避免用户必须切换 Tab 才能扫码
