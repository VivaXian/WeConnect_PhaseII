# PhaseII_Scope_Notes.md

> 用途：给 VS Code / AI 设计模型理解 WeConnect Phase II 范围、角色规则、功能边界与设计约束。  
> 边界：本文件描述 Phase II 需要考虑什么，但不强制规定最终 IA。

---

## 1. Phase II 范围摘要

Phase II 是在 Phase I 基础上的延展设计，目标是复用既有开发和用户心智。

Phase II 聚焦：
- 用户角色与权限差异
- 设备维度的信息扩展
- 合同 / Business Entitlement 信息
- PM 保养计划、结果与风险提醒
- 工单分类与动作
- 消息中心作为提醒归档与回看区域

Phase II 不应变成：
- 完整资产管理 dashboard
- D365 镜像前台
- 后台审批系统
- 脱离设备的合同中心

---

## 2. 触点边界

### WeConnect 小程序
客户触点，负责：
- 客户可见信息
- 客户侧提醒
- 设备、报修、工单、合同、PM、风险信息展示
- 角色化可见控制
- 消息回看与跳转

### D365 / 内部系统
内部触点，负责：
- 主数据来源
- Super User 审批 / 权限配置
- 合同字段维护
- PM 计划来源
- PM 结果回写
- 消息触发逻辑
- 短信发送
- 试点客户数据配置

不要把内部审批流设计进小程序。

---

## 3. 用户角色

只保留两类角色。

### 3.1 普通用户
可见：
- 本人相关报修记录
- 本人相关工单
- 本人相关 / 常用设备
- 相关 PM 结果
- 相关 PM 风险提醒

不可见：
- 合同信息
- 合同到期提醒
- PM 月度计划总览
- 全院设备列表
- 全渠道报修历史

### 3.2 Super User
可见：
- 授权医院 / 院区设备范围
- 权限范围内报修记录
- 权限范围内工单列表
- 设备维度合同信息
- 设备维度 PM 状态
- PM 月度计划
- 风险设备
- 设备维度全渠道报修历史

重要说明：
- Super User 不是只看“我的报修”。
- Super User 是管理视角，但仍需移动端轻量体验。

---

## 4. 推荐导航方向

当前倾向：两类角色共用同一套底部导航。

建议底部导航：
1. 报修
2. 设备
3. 工单
4. 我的

理由：
- 报修：延续 Phase I 报修与报修进展心智。
- 设备：Phase II 核心对象入口，尤其服务 Super User。
- 工单：工单对象与签字、评价、PDF、邮箱等动作承载页。
- 我的：账户、权限、设置、消息中心入口。

不要使用以下作为底部 tab：
- 首页
- 服务进展
- 待签工单
- 合同

---

## 5. 功能范围

## 5.1 报修
目的：延续 Phase I 报修入口和报修进展查看。

普通用户：
- 扫码报修
- 输入设备号报修
- 本人报修记录
- 本人报修进展

Super User：
- 保留报修入口
- 权限范围内报修记录
- 状态筛选
- 可有轻量摘要，但不做重 dashboard

不要在这里放完整合同详情或完整 PM schedule。

---

## 5.2 设备
目的：Phase II 核心对象视图。

普通用户：
- 我的设备
- 最近 / 常用设备
- 设备基础信息
- 设备下本人报修记录
- 相关 PM 结果 / 风险提醒
- 不显示合同信息

Super User：
- 授权范围 / 全院设备
- 快速筛选：全部设备、即将到期、本月保养、风险设备、服务中
- 设备卡片只突出一个主状态，不要堆所有字段
- 设备详情可包含：基础信息、合同信息、报修历史、PM 计划 / 状态、风险状态

PM schedule 必须在设备页有强感知，不能只放消息中心。

---

## 5.3 工单
目的：工单对象与收尾动作页，不只是待签。

工单类型：
- 维修
- 保养
- 安装
- FCO

工单状态 / 动作：
- 待签字
- 已签字
- 请求已失效
- 评价服务
- 下载 PDF
- 发送邮箱

普通用户：
- 本人相关工单
- 签字 / 评价 / PDF / 邮箱动作

Super User：
- 权限范围内工单
- 数量可能较多，分类和筛选很重要

推荐筛选：
- 全部
- 待处理
- 待签字
- 已完成
- 已失效
- 工单类型
- 时间范围
- 医院 / 院区（Super User）

---

## 5.4 我的
目的：账户与工具收口。

包含：
- 账户信息
- 权限状态
- Super User 授权范围说明（如适用）
- 常用设备 / 备注管理
- 消息中心入口

由于微信小程序顶部有固定控件，不要依赖顶部铃铛作为消息中心主入口。

---

## 6. 合同信息

合同信息仅 Super User 可见。

原则：
- 合同信息必须绑定设备。
- 不设计独立合同 tab。
- 不给普通用户展示合同信息。

可展示字段：
- 合同开始日期
- 合同结束日期
- 合同类型

不展示：
- SPN / 特殊权益条款

已知规则：
- 合同结束日期面向客户展示可能需要 6 个月缓冲期。
- 合同到期提醒可能有独立触发时间。
- 不要默认这两条规则是同一件事。

设计重点：
- 帮助 Super User 理解设备保障状态。
- 支撑续保关注。
- 使用客户可读文案，不直接暴露内部术语。

---

## 7. PM / 保养

PM 不应仅作为消息存在。

PM 至少包含三类信息：
1. 计划信息
2. 结果信息
3. 风险信息

### Super User
需要：
- 月度 PM 计划感知
- 本月需要保养的设备
- 设备详情中的 PM 状态
- PM 风险设备

PM schedule 应出现在：
- 设备 tab 快速筛选：本月保养
- 设备列表结果
- 设备详情 PM 区块
- 消息中心仅作为提醒 / 归档

### 普通用户
需要：
- 相关 PM 结果通知
- 相关 PM 风险提醒

普通用户不看完整 PM 月度计划。

---

## 8. 消息中心

消息中心应该存在，但不作为底部 tab，也不主要依赖顶部铃铛。

推荐位置：
- 我的页中
- 业务页面可显示局部提醒摘要

角色：
- 弱提醒归档
- 回看区域
- 不是 PM schedule 或合同信息主入口

消息类型：
- 合同到期提醒（仅 Super User）
- PM 月计划提醒（仅 Super User）
- PM 结果通知
- PM 风险提醒
- 报修 / 服务节点提醒

能力：
- 已读 / 未读
- 消息类型分类
- 可选批量删除
- 点击消息进入相关页面或设备详情

不要：
- 把消息中心作为 PM 唯一入口
- 把顶部铃铛作为主入口
- 未明确要求前，不放权限到期提醒

---

## 9. 筛选策略

复用 Phase I 筛选心智。

### 报修
基础筛选：
- 搜索：设备 / 医院
- 报修状态：已报修、服务中、已完成、已取消
- 时间范围：近三个月、近六个月、近一年

Super User 可增加：
- 医院 / 院区
- 设备名称
- 服务状态

### 工单
基础筛选：
- 搜索：设备 / 医院
- 工单状态：待签字、已签字、请求已失效、已完成
- 工单类型：维修、保养、安装、FCO
- 时间范围：近三个月、近六个月、近一年

Super User 可增加：
- 医院 / 院区
- 是否需要处理
- 设备类型（如需要）

### 设备
普通用户：
- 搜索设备名称 / 医院 / 科室
- 最近设备
- 常用设备
- 有服务中记录
- 有风险提醒

Super User：
- 全部设备
- 即将到期
- 本月保养
- 风险设备
- 服务中
- 医院 / 院区 / 科室
- 合同状态
- PM 状态

### 消息中心
- 全部 / 未读 / 已读
- 消息类型
- 时间范围（如需要）

## 10. UX 原则

- Mobile first
- 复用 Phase I 心智
- 首屏不要过载
- 摘要优先，详情下钻
- 渐进披露
- 一张卡片只表达一个主问题
- 用 chips 做快速筛选
- 用标签表达状态
- 普通用户保持简单
- Super User 提供更强筛选和对象可见性
- 两类角色不应像两个完全不同产品

用户可见推荐用词：
- 报修
- 设备
- 工单
- 保养
- 我的
- 合同信息
- 风险提醒

避免作为用户可见 tab 名：
- 首页
- 服务进展
- 待签工单
- 合同
- Dashboard
- PM schedule
- Business Entitlement

---

## 11. Open Questions

以下不要当作已冻结：
1. 底部导航是否最终确认为 报修 / 设备 / 工单 / 我的
2. 消息中心在“我的”中的具体入口位置
3. 合同可见时间与合同提醒触发时间是否为两套规则
4. PM 风险提醒对象范围
5. Super User 授权设备范围
6. 工单是否需要一级筛选：全部 / 待处理 / 待签字 / 已完成 / 已失效
7. PM schedule 在设备列表与设备详情中分别展示到什么颗粒度
